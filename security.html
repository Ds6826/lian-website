Compliance is architecture.

Policies that live only in application code are policies someone can bypass. Lians pushes isolation, encryption, and access control down into the database and key-management layers — where a bug or a compromised service can't route around them.

Tenant & barrier isolation

The wall is below the app.

Namespace RLS

Every request sets a transaction-scoped app.current_namespace; policies restrict every query to that namespace.

Information barriers

A RESTRICTIVE RLS policy keyed on barrier_group denies cross-barrier reads even for the table owner — verified in CI against a non-superuser role.

Non-superuser posture

Run the app as a non-superuser, non-BYPASSRLS Postgres role. Superusers bypass RLS by design; Lians documents and tests against this.

Compromise-resistant

A compromised service with valid DB credentials still cannot read across a wall — the policy runs under the owner's security context.

Access control & operations

Scoped, audited, observable.

Scoped API keys

Namespace-scoped with read / write / admin scopes, checked on every request; admin audit endpoints also require X-Admin-Secret.

RBAC roles

owner / analyst / compliance / readonly expand to scope sets at auth time.

SSO

OIDC / SAML via gateway forward-auth.

SIEM streaming

Stream audit events to Splunk, Datadog, or Elastic.

Reliability

Idempotency keys prevent duplicate writes on retries; /livez and /readyz probes prevent cascading failures; per-key rate limiting.

Air-gap mode

AIRGAP_MODE hard-fails any config that would send data to an external service.
