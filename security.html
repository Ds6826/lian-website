Compliance is
architecture.

Policies that live only in application code are policies someone can bypass. Lians pushes isolation, encryption, and access control down into the database and key-management layers — where a bug or a compromised service can't route around them.

Tenant & barrier isolation

The wall is below the app.

Namespace isolation and information barriers are enforced by PostgreSQL row-level security, not application checks.

  • Namespace RLSEvery request sets a transaction-scoped app.current_namespace; policies restrict every query to that namespace.
  • Information barriersA RESTRICTIVE RLS policy keyed on barrier_group denies cross-barrier reads even for the table owner — verified in CI against a non-superuser role.
  • Non-superuser postureRun the app as a non-superuser, non-BYPASSRLS Postgres role. Superusers bypass RLS by design; Lians documents and tests against this.
  • Compromise-resistantA compromised service with valid DB credentials still cannot read across a wall — the policy runs under the owner's security context.

Access control & operations

Scoped, audited, observable.

  • Scoped API keysKeys are namespace-scoped with read / write / admin scopes, checked on every request; admin audit endpoints also require X-Admin-Secret.
  • RBAC rolesowner / analyst / compliance / readonly expand to scope sets at auth time.
  • SSOOIDC / SAML via gateway forward-auth.
  • SIEM streamingStream audit events to Splunk, Datadog, or Elastic.
  • ReliabilityIdempotency keys prevent duplicate writes on retries; /livez and /readyz probes prevent cascading failures; per-key rate limiting.
  • Air-gap modeAIRGAP_MODE hard-fails any config that would send data to an external service.
