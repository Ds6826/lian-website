The memory control
plane, explained.

Lians is regulated AI memory infrastructure. Every fact is bitemporal, every change is admitted through a policy gate, and every event lands in a tamper-evident hash chain — so you can prove what an agent knew, when it knew it, and that the record was never altered.

Five layers, one API.

Bitemporal model

Ask what was true,
not just what is stored.

Every fact carries two timelines: event time (when it happened in the business) and system time (valid_from / valid_to — when Lians believed it). Recall runs through a validity gate: present queries return facts where valid_to IS NULL; point-in-time queries return facts where valid_from ≤ as_of < valid_to.

That means recall_at(...) reconstructs exactly what an agent knew on any past date — the foundation for backtests, snapshots, and lineage.

Recall pipeline

Hybrid search,
validity-gated.

BM25 keyword scoring and semantic cosine similarity are blended, then adjusted by recency decay, then passed through the validity gate so superseded versions never reach the model.

The benchmark result: 0 stale facts in top-5 recalls versus 4/4 for an accumulate-everything store, with 100% supersession accuracy. See the eval →

Supersession pipeline

Revised facts close cleanly.

When a new fact arrives, Lians decides whether it supersedes, confirms, or adds to what is already known — in three stages, deterministic first.

  • Stage 1 — Metadata overlapCandidate and existing facts are matched on structured keys (ticker, metric, subject) to find what could conflict.
  • Stage 2 — Deterministic decisionA rule engine classifies the relationship: SUPERSEDES, CONFIRMS, or ADDS — no model required, fully reproducible.
  • Stage 3 — Optional LLM adjudicationFor paraphrase-level conflicts, an optional Claude Haiku stage resolves ambiguity (SUPERSESSION_LLM_STAGE).
  • ResultThe superseded version's validity window closes; it survives for audit but is excluded from present recall.

Tamper-evident audit chain

Proof built into every write.

Each event is appended to a SHA-256 hash chain — every row commits to the hash of the previous one, satisfying SEC 17a-4 / FINRA 4511 append-only requirements.

Anyone can verify the chain: GET /v1/admin/audit/verify returns {"chain_valid": true} or detects exactly where a record was altered. Examiners can export the full log (GET /v1/admin/audit/export, NDJSON or CSV) or reconstruct the complete knowledge state at any past timestamp (GET /v1/snapshot?as_of=…). With WORM mode, the audit object store is object-locked for 17a-4 retention.

Flagship · Memory admission control

Govern what is allowed into memory.

Most memory layers write whatever the agent hands them. Lians puts a policy gate in front of the store — the differentiator no competitor offers.

  • Sensitive-data detectionPII, PHI, and MNPI are classified on the way in, so regulated content is handled by policy rather than by accident.
  • Source-trust validationWrites are scored against source trust so low-trust inputs don't silently become durable facts.
  • Prompt-injection quarantineSuspicious instructions are held out of recall until reviewed, instead of poisoning future context.
  • Review queue + modesHigh-risk writes route to a review queue. Run admission in off, monitor, or enforce mode (ADMISSION_MODE).
